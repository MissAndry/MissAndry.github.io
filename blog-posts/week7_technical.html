SQLInjection

SQL injection is one of the most classic hacks for SQL databases. Basically, hackers take advantage of weaknesses in SQL statements to manipulate parts of the database they're not meant to see. Since a lot of online databases include at least an email address (and in many cases, more sensitive information like credit card numbers), it can have pretty serious reprecussions.

This is an example of an unsecure SQL statement (from Wikipedia):

statement = "SELECT * FROM users WHERE name ='" + userName + "';"

It's pretty clear that the developer intended this to return all the information associated with a certain username (given by the user), but someone could insert "' OR '1'='1" instead of a username to make this statement:

SELECT * FROM users WHERE name='' OR '1'='1';

Since '1' is always going to equal '1' (thanks, math), the above statement will always return "true." Consequently, the malicious user will recieve the entirety of the users table. In the same type of unsecure entry field, an attacker can insert a DROP TABLE command, meaning that all the table information would be permanently lost.

Luckily, there are a few simple ways to avoid SQL injection:

  • Parameter binding
  • Pattern check
  • Escaping, and
  • Database permissions

With parameter binding, a developer can minimize user input. The variable used in the statement (like userName, above) is modified into a placeholder that won't accept an arbitrary SQL statement. Instead, it only deals with a given, developer-controlled type of value. As a result, the malicious SQL code would be treated as incorrect input.

Pattern check is similar to parameter binding. It checks to see if the given input matches a certain type - if the input is supposed to be an integer, float, or boolean. Strings that are supposed to follow a strict pattern (like a date) can also be checked. Since the malicious code is unlikely to match the pretedermined pattern,

Escaping refers to making the SQL statement a "string literal" - for instance, by replacing all the single quotes with two single quotes. The malicious SQL code won't fit into the string literal SQL statement, and subsequently won't be executed. Escaping is more error-prone, however, and should be used in conjunction with something like parameter binding.

Finally, a developer could implement database permissions to reduce the effects of maliciously inserted SQL statements. Database permissions can mitigate which tables are accessible by certain statements, so while a SQL Inject statement would pass, it wouldn't affect all tables in one swoop.

These preventative measures are best used together. No one method is going to give you perfect security against malicious coders, but at least together they can minimize the effects!
